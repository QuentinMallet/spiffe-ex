# Elixir Development Template (with mix2nix)

Comprehensive Nix flake template for Elixir application development with Mix, mix2nix for reproducible builds, testing, and release packaging.

## Features

- ✅ **Elixir and Erlang** latest stable versions from nixpkgs
- ✅ **mix2nix** for reproducible, offline builds with locked dependencies
- ✅ **Mix build system** for dependency management and builds
- ✅ **Testing support** with ExUnit and code coverage
- ✅ **Type checking** with Dialyzer
- ✅ **Language server** (elixir-ls) for IDE integration
- ✅ **Release packaging** with Mix releases
- ✅ **Documentation generation** with ExDoc
- ✅ **Phoenix support** ready (install Phoenix as dependency)
- ✅ **Fully declarative and reproducible** builds with Nix

## What is mix2nix?

`mix2nix` generates a Nix expression (`mix.nix`) from your `mix.lock` file, similar to how `bundix` works for Ruby or `yarn2nix` for JavaScript. This enables:

- **Reproducible builds**: Dependencies are locked in `mix.nix` and fetched from Nix store
- **Offline builds**: No network access needed after `mix.nix` is generated
- **Nix cache integration**: Dependencies can be cached and shared across builds
- **Declarative dependency management**: All dependencies tracked in version control

## Quick Start

### 1. Initialize Your Project

```bash
# Create new project from template
nix flake init -t github:yourusername/machines_conf#elixir

# Or use a specific name
nix flake new my-elixir-app -t github:yourusername/machines_conf#elixir
cd my-elixir-app
```

### 2. Enter Development Environment

```bash
# Start the development shell
nix develop

# Install Hex and Rebar (first time only)
mix local.hex --force
mix local.rebar --force
```

### 3. Create Your Application

```bash
# For a standard application
mix new my_app

# For a supervised application
mix new my_app --sup

# For a Phoenix web application
mix phx.new my_app --no-ecto  # Without database
mix phx.new my_app             # With Ecto/PostgreSQL
```

### 4. Generate mix.nix (Required for Nix Builds)

After fetching dependencies, generate the Nix dependency manifest:

```bash
# Fetch dependencies
mix deps.get

# Generate mix.nix from mix.lock
mix2nix > mix.nix

# Or use the helper script
nix run .#generate-mix-nix

# Commit mix.nix to version control
git add mix.nix
git commit -m "Add mix.nix dependency manifest"
```

**Important**: Uncomment `mixNix = ./mix.nix;` in `flake.nix` packages.default after generating `mix.nix`.

### 5. Development Workflow

```bash
# Fetch dependencies
mix deps.get

# Run tests
mix test

# Run tests with coverage
mix test --cover

# Format code
mix format

# Type check with Dialyzer (first run is slow)
mix dialyzer

# Start interactive shell
iex -S mix

# Build documentation
mix docs
```

### 6. Build Release

```bash
# Build with Nix (reproducible, uses mix.nix)
nix build

# Run the built release
./result/bin/my_app start

# Or run directly
nix run
```

## mix2nix Workflow

### Initial Setup

1. **Create/Update dependencies** in `mix.exs`
2. **Fetch dependencies**: `mix deps.get`
3. **Generate mix.nix**: `mix2nix > mix.nix` or `nix run .#generate-mix-nix`
4. **Commit mix.nix**: `git add mix.nix && git commit -m "Update dependencies"`
5. **Enable in flake.nix**: Uncomment `mixNix = ./mix.nix;` in packages.default

### Adding/Updating Dependencies

Every time you modify dependencies in `mix.exs`:

```bash
# 1. Update mix.exs with new dependencies
# 2. Fetch new dependencies
mix deps.get

# 3. Regenerate mix.nix
mix2nix > mix.nix

# 4. Commit both mix.lock and mix.nix
git add mix.lock mix.nix
git commit -m "Update dependencies: add xyz package"

# 5. Build with Nix to verify
nix build
```

### Helper Script

The template includes a helper script for generating `mix.nix`:

```bash
# Run from project root
nix run .#generate-mix-nix

# This is equivalent to:
# mix2nix > mix.nix
```

## Project Structure

```
.
├── flake.nix          # Nix flake configuration
├── mix.nix            # Generated by mix2nix (commit this!)
├── lib/               # Application source code
│   └── my_app.ex
├── test/              # Test files
│   ├── test_helper.exs
│   └── my_app_test.exs
├── mix.exs            # Mix project configuration
├── mix.lock           # Dependency lock file (commit this)
└── README.md          # Project documentation
```

**Important files to commit**:
- `mix.exs` - Project configuration
- `mix.lock` - Mix dependency lock file
- `mix.nix` - Nix dependency manifest (generated by mix2nix)

```bash
# Run all tests

# Run specific test line
mix test test/my_app_test.exs:42

# Run with coverage report
```

```bash

```

### Code Quality

```bash

```

## Phoenix Web Applications

This template supports Phoenix! To create a Phoenix app:

```bash
# Enter dev shell
nix develop

# Install Phoenix
mix archive.install hex phx_new

# Create Phoenix app
mix phx.new my_app

# Generate mix.nix for Phoenix dependencies
cd my_app
mix deps.get
mix2nix > mix.nix

# Start Phoenix server
mix ecto.create  # Create database
mix phx.server   # Start server at http://localhost:4000
```

### Phoenix with Database

If using PostgreSQL or other databases, uncomment the database client in `flake.nix`:

```nix
# In devShells.default.buildInputs
postgresql  # For PostgreSQL
# sqlite      # For SQLite
```

## Configuration

### Change Elixir Version

Edit `flake.nix`:

```nix
# Use specific Elixir version
beamPackages = pkgs.beam.packages.erlang_26.elixir_1_16;  # Elixir 1.16 + Erlang 26
# beamPackages = pkgs.beam.packages.erlang_25.elixir_1_15;  # Elixir 1.15 + Erlang 25
# beamPackages = pkgs.beam.packages.erlang.elixir_1_14;     # Elixir 1.14 + Erlang (default)
```

### Change Erlang Version

```nix
# Use specific Erlang version
beamPackages = pkgs.beam.packages.erlang_26;  # Erlang 26
# beamPackages = pkgs.beam.packages.erlang_25;  # Erlang 25
```

### Add Development Tools

Add to `devShells.default.buildInputs` in `flake.nix`:

```nix
buildInputs = with pkgs; [
  elixir
  erlang
  mix2nix

  # Add your tools here
  nodejs     # For Phoenix assets
  postgresql # For database
  redis      # For caching
];
```

## Testing

### Unit Tests with ExUnit

```elixir
# test/my_app_test.exs
defmodule MyAppTest do
  use ExUnit.Case
  doctest MyApp

  test "greets the world" do
    assert MyApp.hello() == :world
  end
end
```

### Property-Based Testing

Add StreamData to `mix.exs`:

```elixir
{:stream_data, "~> 0.6", only: [:test, :dev]}
```

```elixir
use ExUnitProperties

property "list reversal" do
  check all list <- list_of(integer()) do
    assert Enum.reverse(Enum.reverse(list)) == list
  end
end
```

### Test Coverage

```bash
# Run with coverage
mix test --cover

# View coverage report
open cover/excoveralls.html
```

## Deployment

### Mix Release

The template builds Mix releases by default:

```bash
# Build release
nix build

# Deploy to server
scp -r result/* server:/opt/my_app/

# Start on server
ssh server '/opt/my_app/bin/my_app start'
```

### Docker (Optional)

Create `Dockerfile`:

```dockerfile
FROM nixos/nix:latest

COPY . /app
WORKDIR /app

RUN nix build

CMD ["/app/result/bin/my_app", "start"]
```

## Troubleshooting

### mix.nix Not Found

Generate it first:

```bash
# Make sure mix.lock exists
mix deps.get

# Generate mix.nix
mix2nix > mix.nix

# Or use helper
nix run .#generate-mix-nix
```

### Dependencies Won't Compile

Make sure you have the required system libraries:

```nix
# Add to buildInputs in packages.default
nativeBuildInputs = with pkgs; [
  gcc
  gnumake
  openssl
  zlib
];
```

### Outdated mix.nix

If builds fail after updating dependencies:

```bash
# Regenerate mix.nix
mix deps.get
mix2nix > mix.nix

# Rebuild
nix build
```

### Dialyzer is Slow

First run builds PLT (Persistent Lookup Table):

```bash
# This is normal and takes 5-10 minutes
mix dialyzer

# Subsequent runs are much faster (seconds)
```

### Mix Archive Not Found

Install archives in dev shell:

```bash
mix local.hex --force
mix local.rebar --force
mix archive.install hex phx_new  # For Phoenix
```

### Port Already in Use (Phoenix)

Change port in `config/dev.exs`:

```elixir
config :my_app, MyAppWeb.Endpoint,
  http: [port: 4001],  # Changed from 4000
  # ...
```

### mix2nix Command Not Found

Make sure you're in the nix develop shell:

```bash
nix develop
mix2nix --version  # Should show version
```

## Why mix2nix?

**Reproducibility**: Traditional Mix builds fetch dependencies from the internet at build time, which can fail if packages are removed or if you're offline. mix2nix locks all dependencies in `mix.nix`, ensuring builds work identically across machines and over time.

**Speed**: Once dependencies are in the Nix store, builds are much faster since they don't need to be refetched or recompiled.

**Caching**: Nix can cache and share built dependencies across projects and machines.

**Declarative**: All dependencies are explicitly declared in `mix.nix`, making it clear what your project depends on.
